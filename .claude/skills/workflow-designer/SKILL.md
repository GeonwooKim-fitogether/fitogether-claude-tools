# 워크플로 디자이너 (workflow-designer)

워크플로(업무 절차·시스템 흐름·파이프라인·자동화)를 **빠짐없이 설계**하고 **시각적으로 보여주는** 스킬.
별도 도구 없이 동작한다(Mermaid는 텍스트로 생성). `superpower`(스펙 우선 규율)와 함께 쓰면 시너지가 난다.

## 언제 쓰나
- 시스템/기능의 흐름을 짜고 있는데 **빠진 단계가 없는지 확인**하고 싶을 때
- 그 흐름을 **플로우차트로 보고** 다듬고 싶을 때
- "워크플로 그려줘", "이 프로세스 빠진 거 없나", "플로우차트로 보여줘" 등의 요청

## 워크플로

### 1단계: 흐름을 구조화한다
사용자의 설명을 받아 아래 요소로 정리한다. 모호하면 **먼저 질문**한다(추측 금지).
- **트리거/시작점** — 무엇이 이 흐름을 시작시키나
- **단계(steps)** — 순서대로. 각 단계의 입력·처리·출력
- **액터/주체** — 각 단계를 누가/무엇이 수행하나 (사람, 시스템, 외부 서비스)
- **결정 분기(decisions)** — 조건과 각 분기의 행선지
- **종료 상태(end states)** — 성공 종료, 실패/취소 종료

### 2단계: 완성도 점검 (핵심)
설계안을 아래 체크리스트로 검토하고, **빠진 항목을 사용자에게 명시적으로 보고**한다.

- [ ] **시작점**이 명확한가 (트리거가 정의됐나)
- [ ] 모든 경로가 **종료 상태**로 끝나는가 (붕 뜬 단계, dead-end 없나)
- [ ] 모든 **결정 분기**의 양쪽(yes/no)이 어디로 가는지 정의됐나
- [ ] **예외/실패 경로**가 있나 (오류, 타임아웃, 거부, 재시도)
- [ ] 각 단계의 **담당(액터)** 이 정해졌나
- [ ] 단계 간 **입력→출력 연결**이 끊기지 않나 (앞 단계 출력이 다음 단계 입력과 맞나)
- [ ] **되돌아가는 루프**(반려·재작업)가 필요한 곳에 있나
- [ ] 사람의 **승인/검토** 지점이 빠지지 않았나

빠진 게 있으면 "⚠️ 누락: …" 형태로 짚고, 채울지 사용자에게 확인한다.

### 3단계: Mermaid 플로우차트로 그린다
정리·점검된 흐름을 Mermaid `flowchart` 로 작성한다.
- 결정은 `{...}`(마름모), 처리는 `[...]`, 시작/종료는 `([...])`
- 분기 엣지에는 조건 라벨(`-->|예|`, `-->|아니오|`)
- 액터가 여러 주체면 `subgraph` 로 묶어 구분
- 예외/실패 경로는 점선(`-.->`)이나 별도 색으로 구분

복잡하면 종류를 바꿔 쓴다: 다중 주체 상호작용은 `sequenceDiagram`, 상태 전이는 `stateDiagram-v2`.

### 4단계: 자동으로 Artifact로 보여준다 (말 안 해도)
플로우차트를 만들거나 수정할 때마다 **사용자가 따로 요청하지 않아도** 시각 결과를 띄운다.

- 가능하면 **Artifact 도구로 렌더**해 미리보기 링크를 바로 제공한다.
  Mermaid를 감싼 최소 HTML로 만들 때는 외부 CDN을 쓰지 말고(아티팩트 CSP가 외부 호스트를 차단함)
  Mermaid 런타임을 인라인으로 넣거나, 마크다운 아티팩트의 ```mermaid 코드블록으로 렌더한다.
- Artifact 도구를 못 쓰는 환경이면, repo에 `.mmd`/마크다운으로 저장하고 경로를 알려준다
  (GitHub에서 ```mermaid 코드블록이 자동 렌더됨).

### 5단계: 반복
사용자 피드백을 받아 1~4단계를 돈다. 완성되면 워크플로를 문서(예: `/plan/` 또는 `docs/`)로 저장할지 물어본다.

## 원칙
- **추측보다 질문**: 흐름이 불명확하면 단계를 지어내지 말고 묻는다.
- **누락은 반드시 표면화**: 점검에서 빈 곳은 조용히 넘기지 말고 명시한다.
- **항상 보여준다**: 흐름을 바꾸면 그림도 같이 갱신해 자동으로 띄운다.
