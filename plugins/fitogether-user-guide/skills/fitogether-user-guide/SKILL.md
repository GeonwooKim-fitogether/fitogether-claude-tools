# Fitogether 제품 유저 가이드 생성 스킬

핏투게더의 모든 제품 가이드가 같은 기준, 같은 품질로 나오도록 하는 스킬이다.
산출물은 **A4 세로 9페이지 내외의 PDF**이며, 디자인은 `assets/example-guide.html`(iPad Sunshield 가이드)의
디자인 시스템을 그대로 재사용한다. 새로운 디자인을 만들지 말 것 — 이 스킬의 존재 이유가 "동일 기준, 동일 퀄리티"다.

## 전체 워크플로

```
1. 인터뷰    → references/interview.md 의 질문지로 담당자에게 내용 수집·튜닝
2. 사진 수집  → references/photo-guide.md 의 샷 리스트를 제시하고 사진 수령
3. 콘텐츠 확정 → 수집한 답변을 가이드 문장으로 다듬어 담당자 확인
4. 제작      → 템플릿 HTML 작성 → 이미지 사전 크롭 → WeasyPrint 렌더링
5. 검수      → 페이지를 이미지로 변환해 시각 검수 (서브에이전트 권장) → 수정 → 전달
```

대화형 환경이라면 1~3단계는 한 번에 다 묻지 말고 섹션 단위로 나눠 묻는다.
담당자가 이미 PPT·문서·사진을 제공했다면 인터뷰는 빈칸을 채우는 용도로만 쓴다.

## 1단계 — 인터뷰

`references/interview.md`를 읽고 질문지를 따라 진행한다. 핵심 원칙:

- **지어내지 않는다.** 담당자가 말하지 않은 사양·수치·호환성 정보는 절대 임의로 채우지 않는다.
  빈 항목은 다시 묻거나, 합의 하에 해당 블록을 생략한다.
- **답변을 튜닝해 준다.** 담당자의 거친 메모를 받아 "문제→해법" 구조의 문장으로 다듬어 제안하고 확인을 받는다.
- 비즈니스팀 독자를 전제로, 기술 용어에는 짧은 풀이를 붙인다. (예: "퀵 릴리스 플레이트(원터치 탈착 판)")

## 2단계 — 사진 수집

`references/photo-guide.md`를 읽고, 필요한 컷 목록(샷 리스트)을 담당자에게 보여주며
"이렇게 찍어주세요"라고 요청한다. 예시 묘사와 함께 컷별 용도를 알려주면 회수율이 높다.
사진이 일부 부족해도 진행은 가능하다 — 없는 컷의 이미지 블록은 빼고 레이아웃을 조정한다.

## 3단계 — 콘텐츠 확정

인터뷰 결과를 아래 고정 목차에 맞춰 배치하고 담당자에게 최종 확인을 받는다.

```
표지 → 문서 안내·목차 → 01 제품 개요(문제/해법) → 02 핵심 특징(2~4개)
→ 03 구성품·사양(표) → 04 조립·사용 방법(STEP 1~N) → 05 비즈니스팀 핵심 포인트(가치 3 + FAQ 4~6)
```

마지막 "비즈니스팀 핵심 포인트" 섹션이 이 문서의 차별점이다. 고객(구단·코칭 스태프 등)에게
설명할 때 쓸 한 줄 가치 제안 3개와, 실제 받을 법한 질문으로 FAQ를 구성한다.
FAQ 답변도 인터뷰에서 확인된 사실만 사용한다.

## 4단계 — 제작

### 템플릿 사용

`assets/example-guide.html`을 작업 폴더로 복사한 뒤 **CSS는 그대로 두고 콘텐츠만 교체**한다.
이 파일의 `<style>` 블록이 디자인 시스템 전체다:

- 색: 차콜 `#17191C`, 포인트 그린 `#00A05A`(밝은 배경) / `#00C76F`(어두운 배경), 보조 `#F5F6F7`
- 폰트: Pretendard (assets/fonts/ 에 OTF 동봉 — 템플릿의 @font-face 경로만 맞추면 됨)
- 반복 컴포넌트: `.feature`(특징 카드), `.stephead/.steps-photos/.howto`(조립 단계),
  `table.spec`(사양표), `.value`(가치 카드), `.faq`(FAQ), `.tip`(팁 박스)

특징이 2개면 `.feature` 블록을 2개만 쓰고, 단계가 5개면 STEP 페이지를 늘리는 식으로
**블록 단위로 가감**한다. 새 컴포넌트 발명은 지양하고, 꼭 필요하면 기존 토큰(색·폰트·radius 2.5mm·테두리 .4pt)을 따른다.

### WeasyPrint 함정 — 반드시 지킬 것

이 디자인을 WeasyPrint로 만들며 확인된 함정들이다. 어기면 사진이 잘리거나 페이지가 깨진다.

1. **`object-fit: cover`를 믿지 말 것.** 가로 사진을 세로형 박스에 넣으면 채우지 못하고 여백이 생기거나
   위쪽만 보인다. 해결: **모든 사진을 박스의 가로:세로 비율과 정확히 같게 사전 크롭**한다.
   `scripts/prep_images.py`를 사용한다 (사용법은 파일 상단 주석).
2. **피사체 중앙 정렬.** 크롭 시 피사체가 프레임 중앙에 오게 한다. 피사체가 박스보다 긴 형태면
   박스 높이를 키우거나(인라인 style), 흰 배경 사진은 패딩으로 보정한다.
3. **페이지 높이는 `296.6mm`.** 297mm 정확값을 쓰면 반올림 오차로 페이지가 쪼개져 빈 페이지가 생긴다.
   `.page`에 `page-break-inside:avoid`도 함께 유지한다.
4. **어두운 배경 페이지**(표지·마지막)는 `@page :first { background:#17191C }`,
   `@page lastpg { ... }` + 해당 div에 `style="page:lastpg;"` 패턴을 그대로 쓴다.
5. 원본 PPT나 사진에 "Title:", "Content:" 같은 **플레이스홀더 글자가 남아있으면 흰 사각형으로 지우고** 쓴다.
6. 사진 속 로고·글자가 뒤집혀 있으면 180° 회전해 바로잡는다.

### 렌더링

```bash
pip install weasyprint --break-system-packages   # 최초 1회
python3 -m weasyprint guide.html "<제품명> 유저 가이드.pdf"
pdfinfo "<제품명> 유저 가이드.pdf" | grep Pages    # 의도한 페이지 수인지 확인 (쪼개짐 감지)
```

페이지 수가 의도보다 많으면 어딘가 넘친 것이다. 콘텐츠를 줄이거나 페이지를 분할한다.

## 5단계 — 검수 (생략 금지)

```bash
pdftoppm -jpeg -r 100 "<제품명> 유저 가이드.pdf" pg
```

생성된 페이지 이미지를 **반드시 눈으로 확인**한다. 가능하면 서브에이전트(새로운 시선)에게 맡긴다.
체크리스트:

- 사진이 박스를 가득 채우는가? 피사체(제품)가 잘리지 않고 중앙에 있는가?
- 텍스트 넘침·겹침·푸터 침범이 없는가?
- 플레이스홀더 텍스트, 뒤집힌 로고가 없는가?
- 단계 번호 원과 제목 행이 정렬돼 있는가?

문제를 고치면 **수정된 페이지만 다시 렌더링해 재확인**하고, 한 번의 수정-검증 사이클 후 종료한다.
완성되면 PDF를 사용자에게 전달한다.

## 톤 & 문구 규칙

- 한국어, 존댓말 단문. 마케팅 과장 대신 사실 중심. 강조는 `<b>`로 키워드만.
- 영문 용어는 첫 등장 시 한글(영문) 병기: "툴리스(Tool-less)"
- 문서 메타: 표지에 작성 부서·적용 모델·버전(`v1.0 · YYYY. MM`)·INTERNAL 표기를 유지한다.
- 버전·날짜는 제작 시점 기준으로 갱신한다.
